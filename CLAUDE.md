## Глоссарий инвариантов (самодостаточно)

Чтобы работать по этому файлу, не открывая конспект. Полные формулировки — `konspekt.md §2`.

- **INV-01** — не владеем финансовой истиной.
- **INV-02** — не владеем идентичностью/статусами; только проекция-кэш с источником и свежестью.
- **INV-03** — система на 1С/БСП; связанность — фича; портируемость вне 1С не цель.
- **INV-04** — после «Подана» заявка идёт только вперёд; черновик вне позвоночника; ожидание/доработка — блок, не обратное ребро.
- **INV-05** — результат всегда документ; отказ — тоже документ.
- **INV-06** — у услуги ровно один центр оказания.
- **INV-07** — процедура фиксируется снимком редакции положения; факты живые.
- **INV-08** — истину проверяет сервер; фронт-валидация не гейт.
- **INV-09** — жизненный цикл = КА в коде; статус меняется только через защищённые переходы; не БизнесПроцесс.
- **INV-10** — внешние мутирующие вызовы идемпотентны и выполняются вне транзакции (Outbox).
- **INV-11** — профиль не универсальный контейнер клиентских данных.
- **INV-12** — внешний эффект идемпотентен либо компенсируем; автообратимости нет.
- **INV-13** — положение — декларативная конфигурация, не исполняемый код; не описывает поведение платформы.

## Карта зависимостей

Кто кого имеет право вызывать. Зависимости — только сверху вниз; снизу вверх — только через события/Inbox.

```
Frontend
   │  JSON-RPC (через API-шлюз)
   ▼
Application        (общие модули-оркестраторы; без тяжёлой слоёнки)
   ▼
Domain             (Заявка — Aggregate Root; переходы; доменные события)
   ▼
SPI                (ServiceConnector / PaymentProvider / DeliveryProvider / EventPublisher)
   ▼
Adapter            (реализация плагина — внешняя обработка)
   ▼
External System

Inbox  → Application      (входящие сообщения поднимаются в приложение)
Outbox → SPI              (исходящие эффекты — вне транзакции)
```

Нарушение направления зависимостей — главный разрушитель архитектуры на горизонте лет. Контролируется (см. «Проверяемость»).

## Как читать правила: MUST / SHOULD / MAY

- **MUST** — архитектурный блокер. Нарушение роняет ревью (в идеале — сборку). Привязано к инварианту.
- **SHOULD** — рекомендованная практика. Отступление допустимо, но обосновывается в PR.
- **MAY** — допустимый вариант реализации, на усмотрение.

## Незыблемое (MUST)

- **[MUST · INV-09]** Состояние заявки меняется только через защищённый метод-переход. Прямое присваивание `Заявка.Состояние = ...` где-либо ещё — блокер.
- **[MUST · INV-09]** **Заявка — Aggregate Root.** Изменения её состояния, блокировок и результата проходят только через доменную операцию заявки. Прямая запись дочерних сущностей (`Блокировка.Записать()` и т.п.) в обход — блокер.
- **[MUST]** **Доменный код не зависит от конкретной реализации источника/коннектора/провайдера — только через SPI.** `ИсточникЗУП.ПолучитьКлиента(...)` из доменного модуля — блокер.
- **[MUST · INV-10]** Никаких синхронных внешних вызовов (HTTP/коннектор/платёж) внутри транзакции 1С. Только Outbox, вне транзакции.
- **[MUST · INV-08]** Любые данные с фронта перепроверяются на сервере. Фронт-валидация — для UX.
- **[MUST · INV-01/02]** Нет своих таблиц-источников баланса/статусов/ФИО — только проекция-кэш. Запись истины наружу — через шлюз/коннектор, не прямой правкой.
- **[MUST · INV-12]** Мутирующий коннектор без `idempotencyKey` — блокер.
- **[MUST · INV-13]** В положении нет исполняемого кода (формул/скриптов). Условные поля — только ограниченный предикат.
- **[SHOULD · INV-11/13]** Новое поле в профиль/положение — сначала вопрос «а его ли это дом».

## Стандарты реализации (рецепты)

### Внешние вызовы — Outbox `[MUST · INV-10]`

Мутирующий вызов наружу (платёж, эффект коннектора, публикация события) — три такта:

1. **В транзакции** смены статуса: записать намерение в регистр Outbox с зафиксированным `idempotencyKey`, заявку в блок, `ЗафиксироватьТранзакцию()`.
2. **Запуск фонового задания — только при `ТранзакцияАктивна() = Ложь`.** Вложенная транзакция в 1С физически не коммитится (лишь уменьшает счётчик) — фоновое в своём изолированном сеансе намерения не увидит. Если переход вызван внутри чужого транзакционного контекста (`ОбработкаПроведения` хоста и т.п.) — **отложить запуск до реального внешнего коммита** (механизм отложенного выполнения / после-транзакционный менеджер БСП).
3. В фоновое передать **ссылку на намерение** (не сканировать таблицу). Вызов наружу с тем же ключом → ответ фиксируется **короткой транзакцией** (снять/сменить блок, записать `externalRef`/`reasonCode`).
4. **Повторы:** N попыток с экспоненциальной задержкой → исчерпаны → заявка в блок «требует разбора» + эскалация. Конкретные числа — TBD.
5. Регламентное задание — **только fallback**: подбирает зависшие намерения; сканирует, запускается редко. Поллинг как основной механизм — не делать.

### Inbox — входящие сообщения `[MUST · INV-10, симметрично]`

Вебхуки, асинхронные подтверждения оплаты, внешние события приходят **at-least-once** — обработка должна быть **ровно-один-раз**:

1. Единая точка приёма. Каждое сообщение несёт `messageId`. Нет своего — синтезировать: **идентификатор отправителя + бизнес-ключи, гарантирующие уникальность в рамках источника** (напр. `providerCode` + `externalTxId`). Слабый синтез (сумма+дата) → коллизия → потеря сообщения.
2. **В транзакции, ДО чтения регистра — исключительная блокировка на `messageId`:**

```bsl
Блокировка = Новый БлокировкаДанных;
Эл = Блокировка.Добавить("РегистрСведений.Кабиторий_ОбработанныеСообщения");
Эл.УстановитьЗначение("ИдентификаторСообщения", MessageId);
Блокировка.Заблокировать();   // второй поток с тем же messageId встанет в ожидание
```

Без блокировки два параллельных потока оба увидят «записи нет» (никто не закоммитил) → двойное исполнение эффекта.

3. Под блокировкой проверить регистр. **Есть → подтвердить приём, выйти** (дедуп). Нет → обработать + записать `messageId` **в одном коммите**.
4. Эффект (смена блока, фиксация оплаты) — доменной операцией через защищённый переход.

### Вышибала vs Outbox `[MUST]` (§6.3)

Регламентный вышибала (просрочка/неоплата) **не уводит заявку в терминал, пока по ней есть активные/неподтверждённые Outbox-намерения.** Сначала проверь очередь: незавершённое → жди разрешения или инициируй компенсацию, потом терминируй. Иначе split-brain: заявка мертва, а наружу ушло успешно.

### Переходы статуса — защищённые методы `[MUST · INV-09]`

Единственная точка смены `Состояние`; таблица допустимых переходов рядом.

```bsl
// Менеджер/объектный модуль документа Заявка
Процедура ВыполнитьПереход(Заявка, НовоеСостояние, Контекст) Экспорт
    Если НЕ ПереходДопустим(Заявка.Состояние, НовоеСостояние) Тогда
        ВызватьИсключение "Недопустимый переход: " + Заявка.Состояние + " → " + НовоеСостояние;
    КонецЕсли;
    // фиксация перехода + движения регистров + доменное событие
КонецПроцедуры
```

Состояние и блокировка — **две независимые оси** (§6.2).

### Самозабор задачи — управляемая блокировка `[MUST]` (§8)

Пул «кто может, тот и берёт». Наивный `Заблокировать()` повиснет до таймаута (~20 с) и выдаст системное «Конфликт блокировок». Правильно: оптимистичная проверка → захват с двойной проверкой → перевод исключения во внятный UI.

```bsl
// 1) Оптимистично, без блокировки
Если ЗначениеЗаполнено(ЗадачаСсылка.Исполнитель) Тогда
    Возврат Отказ("задачу уже забирает другой оператор");
КонецЕсли;

// 2) Захват + двойная проверка
НачатьТранзакцию();
Попытка
    Блокировка = Новый БлокировкаДанных;
    Эл = Блокировка.Добавить("Задача.<Имя>");
    Эл.УстановитьЗначение("Ссылка", ЗадачаСсылка);
    Блокировка.Заблокировать();                 // проигравший ждёт до таймаута
    Объект = ЗадачаСсылка.ПолучитьОбъект();
    Если ЗначениеЗаполнено(Объект.Исполнитель) Тогда
        ОтменитьТранзакцию();
        Возврат Отказ("задачу уже забирает другой оператор");
    КонецЕсли;
    Объект.Исполнитель = ТекущийИсполнитель;
    Объект.Записать();
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    // НЕ показывать сырое «Конфликт блокировок» — перевести в UI
    Возврат Отказ("задача занята, обновите список");
КонецПопытки;
```

`[MAY]` При высокой конкуренции (десятки операторов на очередь) — семафор: регистр `ЗанятыеЗадачи` с контрольным временем, обновляемым по таймеру; при истечении захват освобождается.

### Услуга-действие: порядок и orphan `[MUST · INV-12]`

Строго: **черновик-артефакт (PDF готов) → эффект коннектором (идемпотентно, вне транзакции) → фиксация «Результат сформирован» короткой транзакцией.** Коннектор упал — стереть черновик, заявка в блок с ошибкой.

**Orphan:** фиксация упала *после* успешного коннектора → запись осталась наружу. Идемпотентность спасает от дублей, не от сирот. `externalRef` — основание обнаружения/компенсации. Saga — TBD (§21); пока минимум — явный блок «требует разбора», не молчаливое закрытие.

### Версии файлов `[MUST]` (§5.4)

Заявка хранит ссылку на **конкретную версию** из БСП «Работа с файлами» (не на контейнер). Одно поле анкеты → одна ссылка на актуальную версию, фиксируется на каждом гейте подачи (Подана, снятие блока доработки). Историю не дублировать — она в БСП.

### Кэш сегментов `[MUST]` (§7)

Витрина каталога **не крутит СКД на лету** — читает регистр `СегментыКлиентов`. Индекс пересобирается фоном (событие `СтатусКлиентаИзменён` / регламент), вне транзакции смены статуса. Живой СКД — только гейт подачи и мониторинг. **Окно неконсистентности допустимо** (§7): клиент видит услугу по старому индексу — гейт подачи не пустит, вернёт внятную ошибку. Синхронную пересборку на каждый вход не городить.

### Анти-ghost очистка дельты `[MUST]` (§5.3)

Серверный валидатор на гейте подачи — два прохода: (1) вычислить граф видимости полей по предикатам и пришедшим триггерам; (2) **вырезать из дельты поля, которые сейчас скрыты**, и только потом фиксировать. Иначе «передумал и скрыл поле» оставит призрачные данные, которые на Акцепте уйдут в профиль.

### Материализация снимка положения `[MUST · INV-07]` (§5.1)

Слияние «Учётная политика + Положение» вычисляется **при публикации редакции** и кладётся **плоским снимком**. В рантайме — никакого слияния и ссылок на глобальные объекты. Иначе правка политики задним числом меняет старые заявки.

### Блокировка заявки — объект

«Увести в блок / снять блок» — это не флаг. Формальные реквизиты:

- `КодБлока` — справочник причин (приостановка / ожидание документов / доработка / ожидание ответа / ожидание оплаты).
- `Причина` — детализация.
- `КтоПоставил`, `КогдаПоставил`.
- `ОстатокТаймераСрока` — сохранён на входе в блок (§6.2).
- `Автоснимаемый` — **не просто флаг:** подразумевает подписчика на событие (Inbox для внешних подтверждений) или регламентный обработчик с блокировкой (для таймеров). Без механизма снятия флаг бессмыслен.
- Таймер блока → эскалация (§6.2).

## Типичные нарушения

| Плохо | Почему | Как правильно |
|---|---|---|
| `НачатьТранзакцию(); HTTPЗапрос(...); ЗафиксироватьТранзакцию();` | INV-10: внешний вызов в транзакции; откат «теряет» ключ → двойное списание | Намерение в Outbox → коммит → вызов вне транзакции |
| `Заявка.Состояние = "Закрыта";` где-то в коде | INV-09: мимо защищённого перехода | `ВыполнитьПереход(Заявка, "Закрыта", ...)` |
| `Блокировка.Записать();` напрямую | INV-09: обход Aggregate Root | Через доменную операцию заявки |
| `ИсточникЗУП.ПолучитьКлиента(...)` из домена | Обход SPI; жёсткая привязка к реализации | Через интерфейс «Источник» (SPI) |
| Коннектор без `idempotencyKey`, повтор по таймауту | INV-12: дубликат эффекта наружу | Ключ генерится до вызова, переживает повтор |
| Inbox обрабатывает без блокировки `messageId` | Гонка двух потоков → двойная обработка | Блокировка `messageId` до проверки регистра |
| `ФоновыеЗадания.Выполнить()` при активной транзакции | Намерение ещё не закоммичено физически | Запуск при `ТранзакцияАктивна() = Ложь` |
| Каталог дёргает СКД-сегмент на каждый GET | Лавина на СУБД | Чтение регистра `СегментыКлиентов` |
| Слияние политики при чтении заявки | INV-07: правка политики меняет историю | Снимок на публикации редакции |
| Внешний вызов на синхронной подписке события | Раздувает транзакцию | Реакция с внешним вызовом — через Outbox |
| Дельта пишется как пришла с фронта | Призрачные поля → порча профиля | Анти-ghost: вырезать скрытые поля на сервере |

## Слой источника данных (extension / standalone)

Код **источник-агностичен** (§18). Доступ к клиентам/статусам/деньгам — через единый интерфейс «Источник», не разбросанными прямыми запросами.

- **extension/** — адаптер читает хост-конфигурацию напрямую.
- **standalone/** — адаптер читает свою базу (наполняется обменом БСП: ЗУП, бух).
- Критичные проверки основания и сегментообразующие атрибуты — синхронная верификация у источника в обход лага проекции.

Доменный код один на обе формы; различаются реализации адаптера.

## Контракты SPI (§16)

Плагины — внешние обработки, самодекларируют роль и возможности.

- `ServiceConnector` — ответ строго `{ success, externalRef?, reasonCode?, reasonDescription? }`. `externalRef`/`reasonCode` **непрозрачны** — не парсить, протаскивать в манифест/UI. Идемпотентен, вне транзакции.
- `PaymentProvider` — единый интерфейс: «инициировано → ожидает / оплачено / отказ» + подтверждение. Идемпотентен, вне транзакции. ЛС — такой же шлюз.
- `DeliveryProvider` — каналы уведомлений; вызов наружу через Outbox.
- `EventPublisher` — публикация событий через Outbox (кандидат, TBD).
- Вход (ЕСИА) и подпись (ГосКлюч/КриптоПро) — **встроены**, не плагины.

## API: JSON-RPC и аутентификация

- **[MUST]** Единая точка входа: один HTTP-сервис `api`, метод POST. Внутри — **программный роутер**: парсит `method`, ищет обработчик в **статическом `Соответствие`** (`method → модуль.процедура`) в одном модуле-диспетчере, оборачивает результат/ошибку в стандартный конверт. **Никакого вызова процедуры по имени, пришедшему с фронта**; неизвестный `method` → ошибка `-32601`. Не плодить URL-привязки в стиле OData.
- **[MUST]** **1С — Resource Server за доверенным API-шлюзом** (Nginx/Kong/Envoy). Шлюз валидирует JWT/OIDC, сессии, CORS и пробрасывает verified claims (ID, роли) в заголовки (`X-Kabitoriy-User-Id`, `X-Kabitoriy-Roles`); 1С доверяет им по mTLS / trusted network. **RS256-валидация на каждый запрос силами 1С запрещена** — синхронная крипта кладёт рабочие процессы кластера. Фронт общается со шлюзом по стандартному OIDC; 1С сырые токены не видит.
- Единый формат ошибки с адресацией на поле: `{ "field": "/анкета/адрес", "code": "INVALID_FORMAT", "message": "..." }`.

## Транзакции, блокировки, журналирование

- **[определение] Короткая транзакция** — только чтение/запись данных 1С: без ожидания пользователя, без внешних вызовов, без длительных циклов обработки. Задаём состав, не миллисекунды.
- **[MUST]** Транзакции короткие; внутри только данные 1С, никаких внешних вызовов.
- **[MUST]** Конкурентный доступ — **управляемые** блокировки (`БлокировкаДанных`), не автоматические.
- **[SHOULD]** Журналирование **структурное** (контекст: заявка, переход, исполнитель, `idempotencyKey`/`externalRef`/`messageId`), не голый текст. Технический след — журнал регистрации; бизнес-след — из регистров (§15).
- **[MUST]** Ошибки коннекторов/шлюзов не глотать: фиксировать `reasonCode`/`reasonDescription`, уводить в осмысленный блок.

## Доменные события (§19)

- Контракт: `eventId`, `eventType`, `aggregateId`, `occurredAt`, `version`, `payload`.
- **[MUST]** Внутренняя реакция (обновить индекс сегментов и т.п.) — синхронная подписка. **Любая реакция с внешним вызовом** (`DeliveryProvider`, `EventPublisher`, биллинг) — **только через Outbox.**
- **[SHOULD]** Реакция, **не влияющая на успех текущей операции**, оформляется событием, а не встраивается синхронно. Иначе синхронные подписки разрастаются.
- **[SHOULD]** Версионирование: изменения аддитивные → потребитель толерантен к незнакомым полям; ломающее → новая `version`. Полная политика совместимости — TBD.

## Фронт (frontend/)

- **[MUST]** Тупой схемо-рендерер. Сервер отдаёт **уже вычисленную дельту-схему**; фронт не вычитает профиль и не сопоставляет семантические типы — рендерит полученное.
- **[MUST]** Профиль для предзаполнения — **один агрегирующий RPC-метод**, не сбор по кускам.
- Транспорт — JSON-RPC через API-шлюз (см. «API»). Валидация — зеркало для UX, не гейт (INV-08).
- Новая услуга = **ноль кода на фронте.** Понадобился фронт-код под конкретную услугу — сигнал, что логика протекла в браузер; вернись к серверной схеме.

## Тестирование (MUST)

Система с жёстким КА + Outbox/Inbox без тестов не живёт. Обязательно покрываются:

- переходы статусов (допустимые/недопустимые; защита от прямого присваивания);
- идемпотентность Outbox и коннекторов (повтор с тем же ключом → один эффект);
- дедуп Inbox (повтор `messageId`, в т.ч. параллельный → одна обработка);
- конкурентный захват задачи (две сессии — один забирает);
- orphan/компенсация (эффект есть, фиксация упала);
- анти-ghost (скрытые поля не уходят в профиль);
- снимок положения (правка политики не меняет старую заявку).

Инструменты: **Vanessa Automation** (BDD-сценарии жизненного цикла заявки), **YAxUnit / xUnitFor1C** (юнит модулей-коннекторов и переходов).

## Проверяемость инвариантов

Цель — исполняемый инвариант, не декларация. Реалистично, по зрелости:

- **На старте — обязательный ручной ревью-чеклист по INV-NN.** Это базовый механизм, на него опираемся.
- **Достижимые автопроверки (вводим сразу), bsl-language-server + кастомные правила SonarQube:**
  - запрет типов `HTTPСоединение`/`WSПрокси`/`WSОпределения` в модулях с суффиксом `_Домен` и в обработчиках `ПередЗаписью`/`ПриЗаписи`/`ОбработкаПроведения` (INV-10);
  - коннекторы — в отдельных модулях `_ИнтеграцияИсполнение`; запрет их вызова из транзакционных контекстов;
  - regex/AST на `.Состояние =` и `["Состояние"] =` вне модуля менеджера `Кабиторий_Заявка` (INV-09);
  - запрет вызова модулей-адаптеров из доменных модулей (обход SPI).
- **Тяжёлое не обещаем:** статика не видит границы транзакций сквозь хелперы. Такие проверки — по мере зрелости BSL LS, без обещания ронять сборку.
